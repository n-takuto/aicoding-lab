## 記事執筆ルール

### 必須

- **`pubDate` には必ず「今日の日付」を入れる**（未来の日付や過去の日付にしない）
- **同日に複数の記事を書く場合は時刻も明記する**（例: `'2026-05-23T20:00:00'`）
  - 理由: 並び順（前後記事ナビ等）を時系列に正しく出すため
- **既存記事を修正したら `updatedDate` を更新する**（`pubDate` は触らない）
  - `pubDate` = 初出公開日（並び順に影響）
  - `updatedDate` = 最終更新日（表示のみ、並び順に影響なし）
  - 例: `updatedDate: '2026-05-24T15:00:00'`
- **`title` は32文字以内**にする（Google検索結果で末尾が切れないため）
  - 検索キーワードは含めつつ、自然な日本語にする
- **`description` は120〜160字**で記事の要約を書く（SEO重要）
  - 短すぎるとSEO弱い、長すぎると検索結果で切れる
- **`H2` 見出しを5〜8個程度**で構成（スキャンしやすく）
- **記事末尾はまとめ + ブログ宣伝**で締める（次回予告は不要）
- **末尾に「次のステップ」or「関連記事」セクションを必ず設置**
  - まとめセクション内（H3）or 独立したセクションとして
  - 他の関連記事への内部リンクを最低1〜2本
  - 絵文字アイコン（📖📥🚀など）で視覚化、1行サマリも付ける
  - 目的: 読者の回遊率UP、SEO評価UP
- **時間に依存する情報には必ず具体的な日付を入れる**
  - ❌ NG: 「現時点」「今は」「最近」
  - ✅ OK: 「2026年5月24日時点」「2026年5月確認」
  - 理由: 記事は何年も残るので、いつの情報か明示しないと読者が混乱する

### トーン

**基本姿勢**: フレンドリーで親しみやすい日本語。「ですます調」ベース。
「初心者でも迷わない」「自分ごとに感じる」を最優先する。

#### ❌ 避ける書き方（硬い・他人事感ある）

| NG | 理由 |
|----|------|
| 〜することができます | 冗長 |
| 〜と言えるでしょう | 評論家口調 |
| 本稿では〜について解説します | 教科書っぽい |
| 以下のような〜が挙げられます | 列挙が硬い |
| ご利用ください / ご確認ください | ビジネスメール風 |
| 〜が可能です | 「〜できます」で十分 |
| 〜される場合があります | 「〜することも」で十分 |

#### ✅ 推奨する書き方（親しみやすい）

| OK | ニュアンス |
|----|----------|
| 〜できます | 簡潔 |
| 〜だと思います / 〜って感じです | 主観を入れる |
| 今回は〜を見ていきます / 〜紹介します | 軽い導入 |
| 例えばこんなのがあります | 自然な例示 |
| 使ってみてください / やってみましょう | 行動を促す |
| 〜ですよ / 〜なんです | 親密感 |

#### その他のテクニック

- 主語に「あなた」を使ってOK（読者に話しかける感じ）
- 「ぶっちゃけ」「ざっくり」「サクッと」「ハマる」など砕けた表現で導入
- **失敗談・困った話を入れて共感を呼ぶ**（「私もここでハマりました」）
- 数字や具体例で抽象度を下げる（「数千トークン程度」「月¥3,000」など）
- 読者の気持ちを先回り（「ここで詰まるかも」「これ気になりますよね」）
- 専門用語は最初に説明を入れる（例: `irm` は Invoke-RestMethod の略 など）
- 絵文字を適度に使う（多すぎはNG、見出しに1個など）
- **英語のネットスラングは使わない**（日本人初心者向けなので）
  - ❌ TL;DR、IMO、FYI、ASAP、etc.
  - ✅ 「サクッと言うと」「個人的には」「参考までに」など日本語に

### 構成

- リード文（記事の目的と読了後の状態を提示）
- 必要な前提（環境・知識）
- 本編（H2ごとに区切る）
- まとめ（次回予告は不要）
- フッターのブログ紹介

### デザイン・レイアウト

- **表（テーブル）には罫線が入る**（global.css で設定済み、Markdownでただ `|` 書けばOK）
- **H2 セクション間に余白がある**（global.css で設定済み、見出しを連発しないこと）
- 文章だけで埋めず、**表・コードブロック・引用・絵文字** で視覚的に区切る
- 比較や対比は **表で見せる** のが鉄則（読みやすさ激変する）
- 長文セクションは中見出し（H3）で分割する

### Claude 関連情報を扱うとき

- **必ず公式ドキュメント（code.claude.com / docs.anthropic.com）を最初に確認**
- バージョンや料金変更が頻繁なので、書く前に最新情報をWeb検索する
- サードパーティ情報は補助として使う

#### Anthropic 関連 URL の使い分け（重要）

**🚨 大原則：このブログは日本人初心者向け。日本語ページがあれば必ず日本語版を使う。** 英語ページに飛ばすのは NG。

| 用途 | 日本語URL（記事で使う） | 英語URL（参考） |
|------|--------------------|----------------|
| **サブスク契約・プラン比較** | `https://claude.com/ja/pricing` | claude.com/pricing |
| **Claude Code 公式ドキュメント** | `https://code.claude.com/docs/ja/` | /docs/en/ |
| **Anthropic API ドキュメント** | `https://docs.anthropic.com` (英語のみ) | - |
| **API（開発者向け）コンソール** | `https://console.anthropic.com` | - |
| **通常のClaude（チャット）** | `https://claude.com` | - |
| **Anthropic 公式お知らせ** | `https://www.anthropic.com/news` | - |

#### 記事にリンクを貼る際の必須チェック

1. **URL を貼る前に、日本語版があるか確認する**（`/ja/` パスや言語切替の有無）
2. **そのリンク先が「ユーザーが期待する内容」と一致してるか確認する**
   - 例: 「プラン契約」と書いて API コンソールに飛ばすのは NG
3. **疑わしい時は実際にブラウザで開いて確認する**

⚠️ 過去にやらかしたミス（再発防止メモ）:
- 「プラン契約は console.anthropic.com」と書いた → 実際は API 用。正しくは claude.com/ja/pricing
- 英語版 pricing ページを貼った → 日本語版があるので日本語版を貼るべき

### 記事執筆フロー（必須）

**叩き台をユーザーに見せる前に、必ず以下を実施する：**

1. 記事を書く
2. **3つの検証エージェントを並列起動**（後述「検証エージェント3体制」参照）
3. 各エージェントの結果を集約 → 記事を修正
4. **修正済みの状態で**ユーザーに確認依頼

→ ユーザーが最初に見る時点で「事実・文体・SEO 全部チェック済み」の状態にする。

### 検証エージェント3体制

並列で3つのエージェント（general-purpose）を起動。各エージェントには以下の指示書を渡す。

#### 🔍 Agent A: 事実検証

```
記事ファイル: <パス>
以下を検証して、❌明確な間違い / ⚠️修正推奨 / ✅問題なし で報告。

【一次ソース（必ずWebFetchで確認）】
- https://code.claude.com/docs/en/ (Claude Code 公式ドキュメント)
- https://code.claude.com/docs/ja/ (日本語版)
- https://docs.anthropic.com (Anthropic API)
- https://claude.com/pricing (料金プラン)
- https://www.anthropic.com/news (公式お知らせ)

【サブページ例】
- /docs/en/setup (セットアップ)
- /docs/en/commands (コマンド一覧)
- /docs/en/skills (Skills)
- /docs/en/memory (CLAUDE.md / メモリ)
- /docs/en/mcp (MCP)
- /docs/en/hooks (Hooks)
- /docs/en/sub-agents (Subagents)
- /docs/en/scheduled-tasks (Routines)
- /docs/en/settings (Settings)

【検証項目】
1. 上記の一次ソースと照合
2. 紹介してるコマンド・API・コードが実在・現役か
3. バージョン情報・料金情報が最新か
4. リンク先（内部・外部）が存在するか
5. **リンク先が記述内容と一致してるか**（例：「プラン契約」リンクが API コンソールに飛んでないか）
6. **日本語ページがある URL は日本語版を使ってるか**（例: `claude.com/ja/pricing`、`code.claude.com/docs/ja/`）
7. サードパーティ情報を断定的に書いてないか（憶測は注記必須）

【ルール】
- 一次ソースに記載がない情報は「断定NG、注記必須」と判断する
- サードパーティブログ（Zenn/Qiita/個人ブログ等）は補助参考のみ。これらに書かれてても公式に裏付けがなければ「未確定情報」扱い
- リンク先は必ず WebFetch で実際の内容を確認する
- 推測で答えない。必ずWebFetchで公式を確認
- 300語以内で報告
```

#### ✍️ Agent B: 文体・トーンチェック

```
記事ファイル: <パス>
プロジェクトCLAUDE.md の「トーン」セクションのルールに準拠してるか検証。

1. NG表現が混入してないか
   - 「〜することができます」「〜と言えるでしょう」「ご利用ください」等
2. 曖昧な時間表現がないか
   - 「現時点」「今は」「最近」等 → 具体的日付に置き換えるべき
3. 読みやすさ
   - 同じ文末の連続（〜ます。〜ます。〜ます。）
   - 1文が長すぎないか（80字以内目安）
   - 1段落が長すぎないか（4-5行以内目安）
4. 専門用語に初出説明があるか
5. 失敗談・共感ポイント・読者への呼びかけがあるか

❌違反 / ⚠️改善余地 / ✅問題なし で分類して300語以内で報告。
```

#### 📋 Agent C: SEO・構造チェック

```
記事ファイル: <パス>
SEOと構造を検証。

1. title が32文字以内（Google検索結果で切れない）
2. description が120〜160字
3. H2 が5〜8個程度（多すぎ・少なすぎNG）
4. 見出し階層が h2→h3 で飛んでない（h2のあとh4はNG）
5. リード文がある（最初の段落が記事の導入になってる）
6. まとめがある（末尾に総括）
7. 表・コードブロック・箇条書きで視覚区切りがある
8. 内部リンク（他記事への参照）が整合してる
9. 画像があれば alt 属性がある

各項目を ❌/⚠️/✅ で評価して300語以内で報告。
```

#### 並列実行のコツ

- 3エージェントを **同時に Agent ツールを呼ぶ**（1メッセージで3つ起動）
- 結果が全部返ってきてから集約・修正
- 修正後はもう一度実行する必要なし（時間・コストの無駄）

## 作業日記（drafts/diary.md）の運用

- **セッション終了時に必ず追記する**
  - その日に何をやったか、どこで詰まったか、どう解決したか、学び・気付き
  - 時系列で書く（最新を下に追加）
  - セクション見出しに日付を入れる（例: `## 2026-05-25（土）`）
  - 失敗談・本音もOK（`drafts/` は gitignore 対象でプライベート）
- 用途: 後日「ブログ記事化する時の素材」「振り返り」
- 文章は記事用に整形しなくてOK。事実と感情を生々しく残す方が価値ある
